Visual C
Anatomy of a Garden Entry
Every section has a purpose — none is decoration
Frontmatter id, type, stack, labels, keywords, symptom, context, score, captured, verified — machine-readable metadata. The index and summary files are built from this. Never loaded without a prior keyword match.
Symptom What the developer observes — the misleading part. "No error is reported" is important context here. This is what gets surfaced in the index for retrieval matching.
What was tried Bulleted list of failed approaches and their outcomes. This is the section that saves the next Claude from repeating the same dead ends.
Root cause What was actually happening — the mechanism, not just the outcome. Explains why, not just what.
Fix Actual code or config. Always specific. Always runnable. "Use X instead" is not a fix.
Why non-obvious The insight. Why would a skilled developer be misled? What makes this a gotcha rather than just a bug?

Visual F
Before & After: Retrieval
The v1 retrieval model vs. the v2 indexed approach
v1 — Before
  • Read GARDEN.md (entire index, all titles)
  • Load the full domain file (all entries in it)
  • Grep within the file for the right entry
  • All entries in a domain load as overhead
  • Cost grows with garden size
  • Title-only matching — no keywords, no symptom
→
v2 — After
  • Read _index/global.md (~80 tokens)
  • Grep domain INDEX.md for keywords + symptom
  • Load exactly the matching entry by path
  • No other entries touch the context window
  • Cost is bounded regardless of garden size
  • Keyword + symptom matching — semantically rich
