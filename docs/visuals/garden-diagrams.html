The Garden Ecosystem
Three layers — from open protocol to community knowledge to private extension

Layer 1 · Foundation
The Protocol
An open specification for structured AI knowledge — entry format, index structure, federation rules, quality lifecycle. Tool-agnostic: any AI assistant can implement it.

The Engine
The platform that runs gardens: validation, CI/CD automation, index maintenance, staleness detection, federation tooling, web app, MCP server. Implements the spec.

Layer 3 · Community Knowledge
Canonical Domain Gardens
Community-curated gardens per technology domain — quality-gated, peer-reviewed, governed. The shared knowledge layer that any team can build on.
🔧 tools-garden (GE-) ☕ jvm-garden (JE-) 🐍 python-garden (PE-) 🌐 frontend-garden (FE-) ⚙️ systems-garden (SE-)

🔒
Layer 4 · Private Extension
Child & Peer Gardens
Teams extend the canonical gardens privately. Private entries fill gaps. Private augmentations add context to public entries. Watch CI monitors upstream for incoming duplicates. The parent never knows the child exists.
🏢 enterprise/private-garden 👤 you/knowledge-garden 🔬 research-org/garden
Diagram 2
How Retrieval Works
Bounded token cost at every tier — no vector database required
Tier 1 — Technology Known "This is a Quarkus CDI problem"

Read _index/global.md
→ Find domain row
→ Grep quarkus/INDEX.md for keywords
→ Load matching entry by path
~1,000–1,500 tokens
Tier 2 — Symptom Known "Silent failure, no error visible"

Read labels/INDEX.md
→ Find symptom label
→ Load label file → candidate list
→ Load matching entries by path
~800–1,400 tokens
Tier 3 — Unknown "No idea what's causing this"

Glob _summaries/ → read all
(100 tokens × N entries)
→ Identify best matches
→ Load 2–4 full entries by path
Bounded regardless of garden size
The key principle: The file is the retrieval unit. Every entry is individually addressable by path. No grep-within-file. No bulk loading. The index narrows; the path delivers.
